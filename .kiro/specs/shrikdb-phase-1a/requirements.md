# Requirements Document

## Introduction

ShrikDB Phase 1A is an event-sourced database system that provides a production-ready, append-only immutable event log with deterministic replay capabilities. The system must be crash-safe, support strong project isolation, and provide full observability. This is NOT a prototype or demo - all components must be production-grade with no mock data or fake implementations.

## Glossary

- **ShrikDB**: The event-sourced database system being developed
- **WAL**: Write-Ahead Log, the append-only immutable event log that serves as the single source of truth
- **Event**: An immutable record with project_id, event_type, payload, sequence number, timestamp, and cryptographic hash
- **Replay Engine**: Component that rebuilds system state deterministically from the WAL
- **Project**: A tenant isolation boundary with independent event sequences
- **Sequence Number**: Monotonically increasing per-project counter ensuring deterministic ordering
- **Event Hash**: SHA-256 hash of event data chained with previous event hash
- **Backend**: The Go-based ShrikDB server providing APIs and event storage
- **Frontend**: The TypeScript/React web interface for interacting with ShrikDB
- **Projection**: Derived state computed from events (can be deleted and rebuilt)
- **Client Key**: Authentication credential for API access

## Requirements

### Requirement 1: Immutable Event Log

**User Story:** As a system architect, I want an append-only immutable event log, so that all system state changes are permanently recorded and auditable.

#### Acceptance Criteria

1. WHEN an event is appended to the WAL, THE ShrikDB SHALL write the event with fsync to ensure durability
2. WHEN the WAL is written, THE ShrikDB SHALL prevent any modification or deletion of existing events
3. WHEN an event is created, THE ShrikDB SHALL assign a monotonically increasing sequence number per project
4. WHEN an event is stored, THE ShrikDB SHALL compute a SHA-256 hash including the previous event hash
5. WHEN a partial write occurs during a crash, THE ShrikDB SHALL detect and recover from the corrupted entry on restart

### Requirement 2: Deterministic Replay

**User Story:** As a database operator, I want to rebuild all system state from the WAL, so that I can recover from data corruption or migrate to new projection formats.

#### Acceptance Criteria

1. WHEN the replay engine processes the WAL, THE ShrikDB SHALL apply events in sequence number order per project
2. WHEN replay completes, THE ShrikDB SHALL produce identical state regardless of how many times replay is executed
3. WHEN projections are deleted and the system restarts, THE ShrikDB SHALL fully rebuild all state from the WAL
4. WHEN replay is in progress, THE ShrikDB SHALL report progress metrics including events processed and estimated completion time
5. WHEN replay encounters an invalid event hash, THE ShrikDB SHALL halt and report the corruption location

### Requirement 3: Project Isolation

**User Story:** As a multi-tenant system administrator, I want strong project isolation, so that one tenant cannot access or interfere with another tenant's data.

#### Acceptance Criteria

1. WHEN an event is appended, THE ShrikDB SHALL validate that the project_id matches the authenticated client's project
2. WHEN querying events, THE ShrikDB SHALL return only events belonging to the authenticated project
3. WHEN computing sequence numbers, THE ShrikDB SHALL maintain independent sequences per project
4. WHEN replaying events, THE ShrikDB SHALL process each project's events independently

### Requirement 4: Production Authentication

**User Story:** As a security engineer, I want production-grade authentication, so that only authorized clients can access the system.

#### Acceptance Criteria

1. WHEN a client authenticates, THE ShrikDB SHALL validate credentials using bcrypt or argon2 hashing
2. WHEN authentication fails, THE ShrikDB SHALL log the attempt with timestamp, client identifier, and reason
3. WHEN authentication requests exceed rate limits, THE ShrikDB SHALL reject requests and return appropriate error codes
4. WHEN a client key is validated, THE ShrikDB SHALL use constant-time comparison to prevent timing attacks
5. WHERE key rotation is required, THE ShrikDB SHALL support multiple valid keys per project simultaneously

### Requirement 5: Frontend Integration

**User Story:** As a frontend developer, I want all UI actions to interact with real backend APIs, so that the system operates without mock data or fake state.

#### Acceptance Criteria

1. WHEN a user performs a write action in the frontend, THE ShrikDB SHALL call the AppendEvent API endpoint
2. WHEN the frontend displays data, THE ShrikDB SHALL derive all state from backend API responses
3. WHEN an event is created, THE ShrikDB SHALL return a real event ID generated by the backend
4. THE ShrikDB SHALL NOT maintain any frontend-only state that is not sourced from the backend
5. THE ShrikDB SHALL NOT use mock data generators or dummy response handlers in production code

### Requirement 6: Observability

**User Story:** As a DevOps engineer, I want comprehensive observability, so that I can monitor system health and diagnose issues in production.

#### Acceptance Criteria

1. WHEN the system logs events, THE ShrikDB SHALL output structured JSON logs with correlation IDs
2. WHEN the metrics endpoint is queried, THE ShrikDB SHALL return events per second, write latency percentiles, and replay duration
3. WHEN the health check endpoint is called, THE ShrikDB SHALL report WAL status, replay status, and overall system health
4. WHEN errors occur in WAL operations, THE ShrikDB SHALL log the error with context including operation type and affected project
5. WHEN authentication fails, THE ShrikDB SHALL log the failure with client identifier and failure reason

### Requirement 7: Crash Safety

**User Story:** As a database administrator, I want crash-safe writes, so that the system can recover correctly after unexpected failures.

#### Acceptance Criteria

1. WHEN an event is written to the WAL, THE ShrikDB SHALL call fsync before acknowledging the write
2. WHEN the system restarts after a crash, THE ShrikDB SHALL scan the WAL for partial writes
3. WHEN a partial write is detected, THE ShrikDB SHALL truncate the corrupted entry and continue from the last valid event
4. WHEN WAL corruption is detected, THE ShrikDB SHALL log the corruption details and prevent further writes until resolved
5. WHEN multiple events are written concurrently, THE ShrikDB SHALL serialize writes to maintain sequence number consistency

### Requirement 8: Production Configuration

**User Story:** As a system administrator, I want environment-based configuration, so that I can deploy the same code to dev, staging, and production environments.

#### Acceptance Criteria

1. WHEN the system starts, THE ShrikDB SHALL load configuration from environment variables
2. THE ShrikDB SHALL NOT store secrets in configuration files or source code
3. WHEN the WAL directory is configured, THE ShrikDB SHALL validate the path exists and is writable
4. WHERE different environments are used, THE ShrikDB SHALL support environment-specific settings for log levels and performance tuning
5. WHEN backup procedures are needed, THE ShrikDB SHALL document the process for WAL backup and restoration

### Requirement 9: Testing and Quality

**User Story:** As a software engineer, I want comprehensive tests, so that I can confidently deploy changes to production.

#### Acceptance Criteria

1. WHEN the test suite runs, THE ShrikDB SHALL execute all unit tests and integration tests successfully
2. WHEN integration tests run, THE ShrikDB SHALL test AppendEvent API with real backend calls
3. WHEN crash recovery is tested, THE ShrikDB SHALL simulate crashes and verify correct recovery behavior
4. WHEN replay determinism is tested, THE ShrikDB SHALL verify that multiple replays produce identical results
5. WHEN performance benchmarks run, THE ShrikDB SHALL measure and report write throughput and replay speed

### Requirement 10: Verification

**User Story:** As a project manager, I want automated verification, so that I can confirm Phase 1A is complete and production-ready.

#### Acceptance Criteria

1. WHEN the verification script runs, THE ShrikDB SHALL execute tests against real API endpoints
2. WHEN verification validates WAL data, THE ShrikDB SHALL inspect actual WAL files without using mocks
3. WHEN frontend actions are verified, THE ShrikDB SHALL confirm that UI operations create real events in the backend
4. WHEN projection deletion is tested, THE ShrikDB SHALL verify that restarting the system rebuilds state perfectly from WAL
5. WHEN verification completes, THE ShrikDB SHALL report pass or fail status for all Phase 1A requirements
